# Core Actions and Forms principles review

## Scope and decision boundary

This is the family evidence matrix for issue #158. It reviews the Actions and Forms inventory from
`data/component-catalog.json` against the accepted Tailwind-first implementation, the public API and
entrypoint contracts, native or Base UI behavior, token usage, Registry source installation, docs,
CLI/MCP projections, fixtures, and contract/accessibility coverage.

The review preserves the published visual language, current variants, sizes, slots, compatibility
aliases, and component responsibilities. It does not add a Core component, runtime axis, product
workflow, visual value, or breaking API migration. Search, password, async, multi-select, date, and
validation-library workflows remain future Core candidates, Pro, or consumer composition according
to the existing catalog and tiering rules.

## Cross-family conclusions

- Core responsibilities remain narrow: actions trigger intent; form controls collect or select one
  value; Field and FormGroup own relationships, not validation policy or submission workflows.
- Package and copied-source modes use the same static recipes, Nerio variables, Tailwind bridge,
  `tailwindCn`, Registry dependency graph, and public server/client split.
- The three corrections are existing-contract repairs. They introduce no new prop, variant, token,
  slot, runtime axis, component, or intentional visual value.
- No additional focused follow-up issue is required from this review. Breaking compatibility cleanup
  remains deferred to the existing 1.0 API-freeze work.
